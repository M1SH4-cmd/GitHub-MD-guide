
# Инструкция по использованию GitHub для студентов

## Что такое GitHub

**GitHub** — это платформа для хранения проектов и совместной работы над ними.  
С ее помощью можно:

- хранить код и документы;
- отправлять задания;
- работать в команде;
- отслеживать изменения в файлах;
- возвращаться к предыдущим версиям проекта.

GitHub работает вместе с **Git** — системой контроля версий.

---

## Что нужно установить

Перед началом работы рекомендуется установить:

1. **Git**  
   Нужен для работы с репозиториями на компьютере.

2. **Visual Studio Code** или другой редактор кода  
   Это необязательно, но удобно.

3. **Аккаунт на GitHub**  
   Нужно зарегистрироваться на сайте GitHub.

---

## Основные термины

### Репозиторий
Папка проекта на GitHub, в которой хранятся файлы, история изменений и другие материалы.

### Commit
Сохранение изменений с кратким описанием того, что было сделано.

### Push
Отправка локальных изменений с компьютера на GitHub.

### Pull
Получение изменений из удаленного репозитория на свой компьютер.

### Branch
Ветка проекта. Позволяет работать над изменениями отдельно от основной версии.

### Clone
Копирование репозитория с GitHub на компьютер.

### Pull Request
Запрос на добавление своих изменений в основной репозиторий.

---

## Регистрация на GitHub

1. Перейдите на сайт GitHub.
2. Нажмите кнопку регистрации.
3. Создайте аккаунт.
4. Подтвердите электронную почту.
5. Запомните логин и пароль.

---

## Создание репозитория

Если преподаватель попросил создать свой репозиторий, выполните следующие действия:

1. Войдите в GitHub.
2. Нажмите **New repository**.
3. Укажите название репозитория.
4. При необходимости добавьте описание.
5. Выберите тип:
   - **Public** — репозиторий виден всем;
   - **Private** — доступен только вам и приглашенным пользователям.
6. Поставьте галочку **Add a README file**, если хотите сразу создать файл описания.
7. Нажмите **Create repository**.

---

## Как загрузить проект на GitHub

### Вариант 1. Через сайт GitHub

Подходит для небольшого количества файлов.

1. Откройте нужный репозиторий.
2. Нажмите **Add file**.
3. Выберите **Upload files**.
4. Перетащите файлы в окно или выберите их вручную.
5. Внизу страницы введите сообщение коммита.
6. Нажмите **Commit changes**.

---

### Вариант 2. Через Git на компьютере

Подходит для полноценной работы над проектом.

#### Шаг 1. Клонирование репозитория

Откройте терминал или командную строку и выполните:

```bash
git clone ССЫЛКА_НА_РЕПОЗИТОРИЙ
````

После этого на компьютере появится папка проекта.

#### Шаг 2. Переход в папку проекта

```bash
cd НАЗВАНИЕ_РЕПОЗИТОРИЯ
```

#### Шаг 3. Добавление файлов в проект

Скопируйте нужные файлы в папку репозитория.

#### Шаг 4. Проверка изменений

```bash
git status
```

#### Шаг 5. Добавление файлов в индекс

```bash
git add .
```

#### Шаг 6. Создание коммита

```bash
git commit -m "Добавлены файлы проекта"
```

#### Шаг 7. Отправка на GitHub

```bash
git push
```

---

## Как обновить проект с GitHub

Если в репозитории появились новые изменения, их нужно скачать на компьютер:

```bash
git pull
```

Эту команду желательно выполнять перед началом работы, чтобы у вас была актуальная версия проекта.

---

## Базовый порядок работы студента

Рекомендуемая последовательность действий:

1. Получить ссылку на репозиторий от преподавателя.
2. Клонировать репозиторий на компьютер.
3. Создать или изменить нужные файлы.
4. Проверить изменения командой:

```bash
git status
```

5. Добавить изменения:

```bash
git add .
```

6. Создать коммит:

```bash
git commit -m "Краткое описание изменений"
```

7. Отправить изменения:

```bash
git push
```

---

## Как правильно писать сообщения коммитов

Сообщение коммита должно коротко и понятно объяснять, что изменилось.

### Хорошие примеры

* `Добавлен отчет по лабораторной работе №2`
* `Исправлены ошибки в программе`
* `Обновлен README`
* `Добавлены изображения в папку assets`

### Плохие примеры

* `Изменения`
* `Работа`
* `123`
* `Ну я сделал`

## Работа с ветками

Иногда преподаватель или командный проект требуют работать не в основной ветке, а в отдельной.

### Создание новой ветки

```bash
git checkout -b имя_ветки
```

Пример:

```bash
git checkout -b lab2-belyakov
```

### Отправка ветки на GitHub

```bash
git push -u origin имя_ветки
```

### Переключение между ветками

```bash
git checkout main
```

или

```bash
git checkout имя_ветки
```

---

## Что делать, если проект командный

Если над проектом работают несколько человек, соблюдайте следующие правила:

1. Перед началом работы всегда выполняйте:

```bash
git pull
```

2. Не редактируйте одновременно одни и те же строки в одном файле без согласования.
3. Пишите понятные коммиты.
4. Если требуется, создавайте отдельные ветки под свои задачи.
5. Не удаляйте чужие файлы без необходимости.
6. Перед отправкой проверяйте, что проект запускается и файлы находятся в нужных папках.

---

## Pull Request: как отправить работу на проверку

Если преподаватель просит сдавать задания через отдельную ветку, тогда после отправки изменений нужно создать **Pull Request**.

### Порядок действий

1. Загрузите свою ветку на GitHub.
2. Откройте репозиторий в браузере.
3. GitHub предложит создать **Pull Request**.
4. Нажмите **Compare & pull request**.
5. Укажите название и описание.
6. Нажмите **Create pull request**.

После этого преподаватель сможет посмотреть ваши изменения и принять их.

## Как оформить структуру учебного репозитория

Пример удобной структуры:

```text
project-name/
├── README.md
├── lab1/
│   ├── report.docx
│   ├── code.cpp
│   └── screenshot.png
├── lab2/
│   ├── report.docx
│   └── code.cpp
└── assets/
    └── images/
```

Такой подход помогает поддерживать порядок в проекте.

## README.md: зачем он нужен

Файл `README.md` — это главная инструкция или описание проекта.
В нем обычно указывают:

* название работы;
* автора;
* группу;
* цель проекта;
* структуру файлов;
* инструкции по запуску;
* другую важную информацию.

### Пример простого README

```markdown
# Лабораторная работа №1

**Студент:** Иван Иванов  
**Группа:** ИС-21

## Описание
В данной работе реализована программа для обработки массива.

## Состав проекта
- `main.cpp` — основной код программы
- `report.docx` — отчет по работе
- `screenshot.png` — результат выполнения

## Запуск
Открыть проект в среде разработки и собрать программу.
```

## Частые ошибки студентов

### Ошибка 1. Забыли сделать `git pull` перед началом работы

Из-за этого могут возникнуть конфликты и проблемы с отправкой.

### Ошибка 2. Слишком общие сообщения коммитов

Нужно писать, что именно было изменено.

### Ошибка 3. Загрузка лишних файлов

Не стоит загружать временные файлы, кэш, лишние архивы и ненужные папки.

### Ошибка 4. Изменения сделаны локально, но не отправлены на GitHub

После `commit` нужно обязательно выполнить:

```bash
git push
```

### Ошибка 5. Работа не в той ветке

Перед началом нужно проверить текущую ветку.

---

## Что такое конфликт и что с ним делать

**Конфликт** возникает, когда два человека изменили один и тот же участок файла, и Git не может автоматически решить, какую версию оставить.

В таком случае нужно:

1. Выполнить `git pull`.
2. Открыть файл с конфликтом.
3. Вручную выбрать правильный вариант.
4. Сохранить файл.
5. Снова выполнить:

```bash
git add .
git commit -m "Решен конфликт"
git push
```

## Рекомендации по сдаче работ

1. Следите за аккуратной структурой папок.
2. Называйте файлы понятно.
3. Пишите осмысленные коммиты.
4. Не откладывайте загрузку работы на последний момент.
5. После отправки проверьте на GitHub, что все файлы действительно загрузились.
6. Поддерживайте `README.md` в актуальном состоянии.

---

## Минимальный набор действий для сдачи задания

Если кратко, то для большинства студентов достаточно запомнить 5 команд:

```bash
git pull
git add .
git commit -m "Добавлено задание"
git push
```

И перед этим один раз клонировать репозиторий:

```bash
git clone ССЫЛКА_НА_РЕПОЗИТОРИЙ
```
